Version 02 Enterprise clinical Dense operational UI
Design objective

Make it feel credible before it feels clever.

The product should look like professional clinical operations software: direct, restrained, information-dense, and built for repeated use. It should avoid decorative styling, oversized cards, playful labels, and visual effects that make the application feel speculative.

02 Modules

Independent workflows.

Each module gets a plain overview page and separate patient-specific pages. Do not combine orientation and findings.

RECORD

Patient Record

Neutral chart source. Summary, history, care journey, FHIR data. Lowest interpretation.

PREOP

Pre-Op Support

Surgical readiness, medication holds, anesthesia handoff, clearance, cited evidence.

TRIALS

Clinical Trials

Eligibility search, criteria parsing, patient packet, source-linked trial rationale.

ACCESS

Medication Access

Price research, support programs, manufacturer assistance, pharmacy options.

05 Layout patterns

Plain structural decisions.

Favor navigation clarity and stable panes over visual flourish. These are the patterns to apply first.

Module Overview

Static orientation, workflow scope, included pages, agent role, and trust boundaries.

Patient-Specific Page

Patient header, fact rail, status summary, evidence table, source details, agent panel.

06 Agents

Agent roles should be explicit.

Separate agent harnesses should feel like separate professional tools, not one generic chat box reused everywhere.

Record Agent

Neutral chart questions. Cites FHIR resources. Avoids workflow-specific recommendations.

Pre-Op Agent

Surgical risk, medication holds, clearance gaps, anesthesia handoff, evidence summaries.

Trials Agent

Searches registries, reads criteria, ranks matches, identifies exclusions to verify.

Access Agent

Drug normalization, pricing, manufacturer support, affordability plan, source links.

07 Rules

What to do next.

Use this as a practical checklist for the redesign pass.

Use

  • Flat enterprise panels with borders, not decorative cards.
  • Short section labels and obvious navigation terms.
  • Dense tables and fact rails for patient pages.
  • Reserved color for clinical state and module identity.
  • Visible source type, date, and confidence where possible.

Avoid

  • Playful pills, cute icons, and marketing copy.
  • Huge empty panels on operational pages.
  • Mixing overview instructions with patient findings.
  • Soft pastel overuse that lowers perceived seriousness.
  • Generic dashboard metrics without clinical purpose.
Recommended direction

More boring, more credible: enterprise clinical operations with dense, sourced, patient-specific views.

Next pass: app shell, module overview pages, patient summary, pre-op briefing
