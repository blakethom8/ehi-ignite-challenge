HHS EHI Ignite Challenge · Technical brief · v0.2

Patient-side EHI harmonization with FHIR-native provenance.

How we turn fragmented patient health data — SMART-on-FHIR pulls, Epic EHI Export TSV, CMS claims, C-CDA documents, and PDFs — into one canonical FHIR R4 record with full Provenance lineage exposed to clinicians and patients. Architecture, prior art, and the worked example we ship for Phase 1.

01 · The problem

EHI is broader than FHIR, and the unsolved layer is harmonization.

HHS legally defines EHI as the HIPAA Designated Record Set (45 CFR 164.501): all electronic patient information a covered entity maintains, in any form. ONC's (b)(10) certification criterion does not mandate FHIR — Epic ships EHI Export as ~552 raw TSV tables; other vendors ship CSV, XML, or scanned PDFs. Roughly half of certified EHRs satisfy the requirement by exporting C-CDAs, not FHIR USCDI.

The patient-side prior art splits cleanly. Connectors are solved — Josh Mandel's health-skillz covers 88,000+ providers via SMART on FHIR; commercial aggregators (1upHealth, Particle Health, Health Gorilla, Zus Health) ship managed FHIR servers behind TEFCA / Carequality / Commonwell pulls; Apple Health does patient-side multi-provider FHIR aggregation. Harmonization is not solved. Existing systems either:

  • preserve per-source slices and don't merge (Mandel's published design — explicit non-decision)
  • merge with audit-only Provenance never exposed to end users (1upHealth, Particle, Zus, the cloud-platform offerings)
  • label per-fact source in a closed product without lineage walking (Apple)
  • operate at enterprise scale without a patient-facing UX (Epic Cosmos, OMOP-on-FHIR institutional warehouses)

And none of them connect document extraction (PDFs, CCDAs, scanned faxes — where most patient-side EHI actually lives) into a clinician-facing FHIR record with per-fact source-coordinate provenance.

Our position: we embed Mandel's connector with attribution, build the harmonization layer above it, and ship a clinician-facing Sources panel that walks the Provenance graph uniformly across structured and unstructured sources. Phase 1 demonstrates the architecture on a worked example; Phase 2 productionizes for multi-vendor coverage.
02 · Architecture

Five layers, immutable boundaries, FHIR R4 as the canonical schema.

┌─────────────────────────────────────────┐ │ Source A: SMART-on-FHIR pull (Cedars) │ │ Source B: Epic EHI Export TSV │ │ Source C: CMS Blue Button claims │ │ Source D: C-CDA documents │ │ Source E: Lab report / EOB PDFs │ │ Source F: AHRQ SDoH (optional) │ └────────────────────┬────────────────────┘ │ ┌────────────▼─────────────┐ │ LAYER 1: INGEST │ per-source adapters → │ raw immutable bronze │ Mandel-shape envelope └────────────┬─────────────┘ │ ┌────────────▼─────────────┐ │ LAYER 2: STANDARDIZE │ every source → FHIR R4 │ silver per source │ USCDI / CARIN BB profiles └────────────┬─────────────┘ + vision extraction sub-layer │ ┌────────────▼─────────────┐ │ LAYER 3: HARMONIZE │ ◀── DIFFERENTIATION │ Cross-source dedup, │ per-fact Provenance │ entity resolution, code │ + UMLS code mapping │ mapping, temporal align, │ + Fellegi-Sunter │ conflict detection │ + LLM extraction └────────────┬─────────────┘ │ ┌────────────▼─────────────┐ │ LAYER 4: CURATE │ SQL-on-FHIR views │ analytic warehouse │ enrichments (RxNorm) └────────────┬─────────────┘ │ ┌────────────▼─────────────┐ │ LAYER 5: INTERPRET │ Context pipeline │ clinician-facing UX │ surgical risk briefing └──────────────────────────┘ Sources panel + Q&A

Each layer is an immutable boundary: data moves down by transformation, never by mutation. Provenance flows through — every harmonized fact in gold knows which silver records it came from, which in turn know which bronze artifact they came from. Three-tier storage: 

BRONZE  per-source envelopes, immutable             ← Mandel-shape
        providers[] × fhir{ResourceType[]} × attachments[]
        + binaries (PDF originals) + page rasterizations + bbox text

SILVER  FHIR Bundles per source, profile-validated  ← Layer 2 output
        one Bundle per (source, patient, ingest_run)
        includes vision-extracted resources with extraction-provenance Extensions

GOLD    cross-source merged FHIR + Provenance graph ← Layer 3 output
        one resource per logical fact
        Provenance edges to silver, transitively to bronze

Layers 4 and 5 already exist in production: a SQL-on-FHIR warehouse (data/sof.db) with RxNorm-driven drug-class enrichments, and a four-stage context pipeline (filter → episode-compress → format → persona) that compresses raw FHIR to ~1,400 tokens of clinician-ready briefing for the surgical pre-op scenario. The architectural payoff of standardizing on FHIR at Layer 2 is that Layers 4 and 5 require no changes to consume harmonized multi-source data.

03 · Layer 3 — the differentiation work

Cross-source dedup, entity resolution, code mapping, conflict detection.

The output of Layer 3 is what a clinician would think of as "the patient" — not five disconnected slices. Sub-tasks and the engine that owns each:

Sub-taskPrimary engineWhy
Patient identity resolutionFellegi-Sunter probabilistic linkage on (name, DOB, address, MRN-when-shared)algorithm unchanged since 1969; Splink, N3C, PCORnet, TriNetX use this
Provider identity resolutionNPPES-anchored entity resolutionleverages existing CMS pipelines
Encounter deduphash on (patient, period.start, location) + temporal proximityhigh precision; false negatives keep both with provenance
Condition mergeUMLS CUI-level merge + temporal envelope unionone Hypertension on the problem list, sourced from N records
Medication episode reconciliationRxNorm-ingredient grouping + active-period intervalsextends single-source episode logic to multi-source
Observation dedup(LOINC, effectiveDateTime, value, unit) hash with UCUM unit normalization firstexact match kills duplicates; near-misses preserve both
Code harmonizationUMLS lookups, scriptsdeterministic; 80% of mappings are table lookups
Temporal alignmentUTC normalize; DocumentReference.context.period.start precedence ruleprevents the "metadata date as clinical date" bug Mandel flagged
Free-text → structuredsmall LLM (Haiku-class) per attachment, schema-constrained outputscripts can't do NLP; LLM is the right tool here
Conflict detectionrule first, LLM-judge for ambiguous cases (build-time, not runtime)rules catch common patterns; LLM-judge for novel cases
Quality scoringrule-based: recency × source authority × completenesspolicy needs to be explicit, not LLM-learned
04 · Provenance — use FHIR-native machinery

Don't invent custom fields. FHIR has the vocabulary.

Six native FHIR mechanisms carry source tracking. The Provenance resource is the load-bearing piece — every cross-source merge emits one with target = merged resource, entity[] = source records, agent = our pipeline (versioned), activity = MERGE / DERIVE / EXTRACT, recorded = timestamp.

Worked example: Hypertension merged from Cedars + Epic EHI

// gold/Condition/harmonized-htn-blake
{
  "resourceType": "Condition",
  "id": "harmonized-htn-blake",
  "meta": {
    "profile": ["http://hl7.org/fhir/us/core/StructureDefinition/us-core-condition"],
    "source": "harmonizer://merge/2026-04-29",
    "tag": [
      {"system": ".../source-tag", "code": "cedars-sinai"},
      {"system": ".../source-tag", "code": "epic-ehi-fixture"},
      {"system": ".../lifecycle", "code": "harmonized"}
    ],
    "extension": [
      {"url": ".../quality-score", "valueDecimal": 0.94}
    ]
  },
  "identifier": [
    {"system": "https://fhir.cedars-sinai.org/Condition", "value": "abc123"},
    {"system": "epic-ehi://Condition", "value": "789xyz"}
  ],
  "code": {
    "coding": [
      {"system": "http://snomed.info/sct", "code": "38341003", "display": "Hypertensive disorder"},
      {"system": "http://hl7.org/fhir/sid/icd-10-cm", "code": "I10", "display": "Essential hypertension"}
    ]
  },
  "subject": {"reference": "Patient/blake-merged"},
  "onsetDateTime": "2018-03-04",
  "clinicalStatus": {"coding": [{"system": ".../condition-clinical", "code": "active"}]}
}

// gold/Provenance/...
{
  "resourceType": "Provenance",
  "target": [{"reference": "Condition/harmonized-htn-blake"}],
  "recorded": "2026-04-29T14:21:00Z",
  "activity": {"coding": [{"code": "MERGE"}]},
  "agent": [{"who": {"display": "ehi-harmonizer v0.3.1"}}],
  "entity": [
    {"role": "source", "what": {"reference": "Condition/cedars-abc123"}},
    {"role": "source", "what": {"reference": "Condition/epic-789xyz"}}
  ]
}

Both source records preserved in the bronze tier with original IDs. One merged record in gold with both identifiers retained, both code systems retained (UMLS-mapped, not chosen-between), quality-scored. A Provenance edge written by our pipeline. No information destroyed at any stage.

05 · Multi-format ingestion

Vision extraction with page-level provenance.

Layer 2 has two paths and uses each where it wins.

Path A — deterministic (structured-format documents)

C-CDA → FHIR via Microsoft FHIR-Converter Liquid templates. HL7 V2 → FHIR via the same converter. Battle-tested, free, no LLM cost. Fall back to LLM only for sections the templates don't cover.

Path B — vision LLM (PDFs, complex layouts, free-text-only documents)

  • Input: rasterized PDF pages plus layout-aware text extraction with bounding boxes (pdfplumber for structured PDFs, tesseract OCR fallback for scanned). Combining vision + ground-truth strings outperforms either alone.
  • Engine: Claude Opus 4.7 or Sonnet 4.6 via the Anthropic SDK with native multi-page PDF support; vision-accurate on tables and forms.
  • Output: schema-constrained JSON via Pydantic / instructor / structured outputs. No free-form text in the production path.
  • Determinism via caching: content-hash → cached extraction. Same PDF in, same extraction out. Re-extract only on prompt or schema-version change.
  • Per-fact provenance: every emitted FHIR resource carries meta.extension(.../extraction) with the extraction model, version, confidence, source attachment reference, and page+bbox locator.

Worked example: creatinine value lifted from a Quest lab PDF

{
  "resourceType": "Observation",
  "code": {"coding": [{"system": "http://loinc.org", "code": "2160-0",
                       "display": "Creatinine [Mass/volume] in Serum or Plasma"}]},
  "valueQuantity": {"value": 1.4, "unit": "mg/dL"},
  "effectiveDateTime": "2025-09-12",
  "meta": {
    "source": "lab-report-pdf://2025-09-12-quest.pdf",
    "profile": ["http://hl7.org/fhir/us/core/StructureDefinition/us-core-observation-lab"],
    "extension": [
      {"url": ".../extraction-model", "valueCoding": {"display": "claude-opus-4-7"}},
      {"url": ".../extraction-confidence", "valueDecimal": 0.97},
      {"url": ".../source-attachment", "valueReference": {"reference": "Binary/quest-2025-09-12"}},
      {"url": ".../source-locator", "valueString": "page=2;bbox=144,302,260,318"}
    ]
  }
}
The demo's strongest moment: a clinician clicks a fact in the patient-journey briefing. The Sources panel walks the Provenance edges. For structured sources it opens the parent FHIR Bundle. For PDF-extracted facts it opens the PDF page with the bounding box highlighted. Same UI, same lineage primitive, working uniformly across structured and unstructured sources. None of the surveyed prior-art systems ships this.
06 · Engine principle

LLMs at the boundaries. Scripts in the middle.

Scripts win:

structured-to-structured mapping (column → FHIR field; SNOMED → ICD-10 via UMLS); deduplication on exact-match keys; quality scoring with explicit policy.

LLMs win:

free-text → structured extraction at ingest (clinical notes → Condition); build-time crosswalk generation (validate against ground truth, then freeze); conflict adjudication and "what changed" narration.

Iterative agents:

quality assurance, not transformation. Post-pipeline auditors sample N patients and flag — they don't rewrite. Crosswalk drift detectors watch for vendor schema changes. Adversarial test-case generators feed CI.

Out of scope:

autonomous LLM merge, runtime free-form prompting per record, LLM-as-final-arbiter on clinical facts. Failure modes are well-documented in the prior art and we want none of them in the pipeline.

Cited precedents validate this posture: FHIR-GPT (Wang et al., PMC 2024) achieves >90% exact-match on medication-statement extraction with constrained prompts; LLMonFHIR (Stanford, arXiv 2402.01711) ships physician-validated FHIR-LLM UX. Both are narrow extractors, not general agents.

08 · Prior art & attribution

We embed; we don't reinvent.

Architecture patterns we adopt as table-stakes — not claimed as novel: medallion bronze/silver/gold (Databricks vocabulary; OHDSI's raw → staging → CDM is the same shape); FHIR R4 + USCDI / CARIN BB profiles as exchange schema; UMLS / OHDSI Athena for vocabulary harmonization; Fellegi-Sunter probabilistic linkage; Microsoft FHIR-Converter Liquid templates for HL7v2 / CCDA → FHIR; SQL on FHIR for the analytic warehouse.

Josh Mandel's open-source stack

Mandel is a SMART on FHIR co-creator and the EHI Ignite Challenge's published technical reference. His four repos cover the full patient-data-to-AI access flow. Phase 1 embeds his connector layer with attribution; Phase 2 productionizes our own implementations of the modules we've extended. Specific dependencies:

  • request-my-ehi (Apache-2.0) — Anthropic-format Claude Code skill that generates the PDF request packet to obtain an EHI Export. Embedded as the patient-side request workflow.
  • health-skillz (MIT, claimed) — Bun + React SMART-on-FHIR connector covering 88,000+ providers; ships a Privacy Redaction Studio with auto-suggested PII clustering. Embedded as the data-collection layer for Phase 1.
  • health-record-mcp (MIT) — first-generation MCP server exposing fetched FHIR data via grep/query/eval tools. We adopt the FHIR-to-plaintext renderer (src/fhirToPlaintext.ts, ~1,500 lines covering USCDI).
  • my-health-data-ehi-wip (MIT, claimed) — Vite app ingesting Epic EHI Export TSV through a numbered 0–8 pipeline, including the heuristic logical-table inference (02-merge-related-tables.ts) we lift verbatim into our Layer 1 Epic adapter.
  • sample_ccdas (CC BY 4.0) — 747 PHI-free C-CDA fixtures across ~12 vendors. We use one fixture as Phase 1's CCDA showcase document.

Specific design moves we adopt verbatim: per-source envelope preservation in bronze; bestEffortPlaintext + originals[] attachment-lift pattern; DocumentReference.context.period.start precedence rule for clinical timing; brand-directory tag-filter syntax.

09 · Open source posture

MIT licensed, public showcase patient, reproducible.

  • The harmonizer code is open source from day one (MIT license).
  • The showcase patient is entirely synthetic — derived from one Synthea ground-truth patient projected into Epic EHI shape and CMS Blue Button claim shape. Zero PHI. Reproducible by anyone who clones the repo.
  • All embedded third-party components are credited with their license and source URL.
  • The FHIR Provenance and Extension URLs we mint are documented and versioned; the schema is inspectable, not opaque.
  • Phase 1 build artifacts (the Synthea patient ID, the synthesized lab PDF, the redacted Epic SQLite dump from Mandel's repo, the CCDA fixture) are all redistributable under their respective licenses.
