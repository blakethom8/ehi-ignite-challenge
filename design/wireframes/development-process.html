Overview For patients Technical Process
Phase 1 development process · No timelines, just structure

How we build it.

Our workflow is built around the part of this project we feel most acutely: getting heterogeneous, real-world patient data into one reproducible corpus before anything else can run. This document describes the stages, who owns what, the decision and quality gates between them, and the meta-patterns we use to keep the work coherent across architecture docs, code, and demos.

01 · How we work together

Roles, decisions, and the document-first principle.

This is a two-person build (Blake + Chief, the AI assistant) augmented by spawned sub-agents for parallel research. We split the work along a clean line: Blake owns domain judgment, real-data acquisition, demo decisions, and outreach; Chief owns code, documentation, automation, and orchestration of research sub-agents. Shared decisions land in writing before they get implemented.

Blake owns

Domain & decisions
  • Real-data acquisition (own portal logins, consent for self/family)
  • Strategic decisions (showcase patient choice, AI-bonus angle, scope cuts)
  • Outreach (Josh email, clinician feedback, future collaborators)
  • Demo voice and narrative tone
  • Final say on what's shipped

Chief owns

Code & orchestration
  • Architecture docs and execution plans (always written first)
  • Implementation: adapters, harmonizer, extraction, UI plumbing
  • Sub-agent orchestration for prior-art research, EHI scope review, etc.
  • Test fixtures, validation, automation
  • Status updates and risk surfacing

Shared

Direction-setting
  • Stage gates (alignment before moving forward)
  • Open questions surface immediately, not batched
  • Architecture changes (Chief proposes, Blake approves)
  • Showcase-patient artifacts (Blake picks, Chief implements)
  • Cuts when scope strains the deadline

Document-first principle

Architecture docs are the source of truth, not the code. Before any non-trivial implementation, the relevant doc gets updated; the commit references the doc section it implements. This is what lets sub-agents and future contributors orient quickly, and what lets us pause and resume across sessions without losing the thread.

Document hierarchy:

  • EXECUTION-PLAN.md — what we're building, success criteria, risks, reviewer pushbacks (the active build plan)
  • architecture/DATA-AGGREGATION-LAYER.md — the technical heart (5-layer pipeline, FHIR-native Provenance, scripts vs LLMs)
  • architecture/CONTEXT-PIPELINE.md — existing Layer 5 (LLM context engineering), unchanged by this project
  • research/strategic-options.md — embed/partner/parallel-build decision and rationale
  • research/josh-mandel/ — full prior-art dossier on Mandel's stack
  • research/harmonization-prior-art.md — broader prior-art survey with citations and reviewer-pushback answers
  • design/wireframes/*.html — these pages, the audience-specific outputs

Sub-agent pattern

For research tasks that would otherwise stall the main thread, we spawn focused sub-agents with self-contained briefs. Each writes its findings to a specific path in research/ and returns a punch-line summary. The main thread synthesizes the findings back into the active execution plan. Pattern used so far for the Mandel GitHub inventory, the LinkedIn synthesis, the live-site walkthrough, the EHI Ignite scope review, and the harmonization prior-art survey.

Decision cadence

  1. Open questions surface immediately. When a decision is required, Chief surfaces it explicitly with options, recommendation, and the cost of deferring. Blake answers; the answer goes into the doc.
  2. Stage gates require explicit alignment. Moving from one stage to the next isn't automatic. A short checkpoint, "are we satisfied with Stage N's outputs," precedes the start of Stage N+1.
  3. Cuts are explicit, not silent. When scope strains, the cut order is stated up front (in EXECUTION-PLAN.md §Risks). We don't quietly drop work — we name what we're cutting and why.
02 · Six stages overview

The build, end to end.

Six stages, ordered by dependency. Stage 1 is the longest and the one we most underestimated; the rest are bounded by it. There are no committed timelines on this page — refer to EXECUTION-PLAN.md for day-by-day if you need it. Here we describe structure, ownership, and definition-of-done.

#StageWhat we produceDefinition of done
1 Data acquisition & corpus building Bronze tier populated with all source artifacts; reference terminologies loaded Anyone with the repo + the documented credentials can rebuild the corpus from scratch
2 Standardization & ingestion plumbing Per-source adapters; Layer 2 transforms producing valid silver FHIR Bundles Every source produces profile-validated FHIR R4; CI passes
3 Harmonization layer (Layer 3) Cross-source merge with Provenance graph; conflict + duplicate + orphan detection Showcase patient passes the 5-artifact test suite end-to-end
4 Vision extraction sub-layer PDF → FHIR Observations with page-level Provenance; deterministic cache Same PDF in produces same extraction out; bbox locator round-trips to UI
5 Application integration Sources panel; PDF page-bbox highlighting; conflict + diff UI; AI-bonus polish End-to-end demo runs without manual intervention; live URL deployable
6 Submission packaging Phase 1 narrative PDF, design wireframes, public GitHub repo, frozen demo URL A judge clones the repo, opens the URL, reads the narrative — and arrives at the same one-paragraph summary we wrote

Per-source acquisition workflow

Each source goes through the same five-step workflow, owned per the role split. The point of structuring it this way is that we can apply it to any new source we add later without re-deciding the process.

  1. Plan the acquisition. One README per source under data/real-world-examples/<source>/README.md capturing: what it is, where it came from, who acquired it, when, what consent applies, what license redistribution is allowed under, and any known gotchas. Shared
  2. Acquire the bytes. Different per source — API call, git clone, manual download, generation script. Whichever path, the result lands in data/real-world-examples/<source>/raw/ with a checksum. Blake for portal logins and personal data; Chief for everything else.
  3. Apply the privacy gate. If the source contains real personal data (Blake's, Devon's, Cedars portal scrapes), it stays out of any public artifact until a redaction profile has been applied — using Josh's redaction studio, with named profiles checked into the repo. Original artifacts never modified; redacted versions land alongside under raw-redacted/. Blake
  4. Document the source format. A short note on what shape the data is in, what fields are present, and any pre-engineered artifacts we plant for the showcase patient. Goes in the per-source README. Chief
  5. Stage to bronze. The cleaned, redacted artifact gets copied (or symlinked) into data/real-world-examples/bronze/<source>/<patient>/ in the canonical bronze layout. This is the only path Layer 1 adapters read from. Chief

Privacy gate is non-negotiable. Any source containing real personal data (Blake's, Devon's, Cedars-portal-derived) gets gitignored at the raw/ level. Only the redacted form is committed. Reproducibility test

The corpus is "done" when a contributor with a fresh checkout can rebuild the bronze tier from the documented sources. Concretely:

✓
data/real-world-examples/_sources/<source>/README.md exists for every source and includes the reproduction recipe.
✓
Open sources (Synthea, Josh's repos, BB sandbox) reproduce automatically via a single make corpus or equivalent.
✓
Personal sources (Blake's, Devon's) are documented but gated — the README explains how Blake/Devon would re-acquire if needed; the redacted form is what's committed.
✓
Generated artifacts (synthesized lab PDF) include the generator script in their _sources/ folder and reproduce deterministically.
✓
Reference terminology snapshots are pinned to specific UMLS / LOINC versions in a reference/VERSIONS.md file.

Showcase patient construction

The showcase patient is fully synthetic, derived from one Synthea ground-truth patient projected into the other source shapes (Epic EHI TSV, CMS BB claims, CCDA, lab PDF). This sidesteps both the privacy concerns of using real Blake/Devon data publicly and the artificial-merge problem of stitching real FHIR to synthetic claims about a different person. Five pre-engineered cross-source artifacts demonstrate each Layer 3 behavior — see EXECUTION-PLAN.md §"Showcase patient construction recipe" for the full list.

Real Blake's Cedars data and (if acquired) Devon's stay available as a live-demo "real connector proof-of-life" sidebar — proves the connector works on real records, but the harmonization story runs on the synthetic showcase patient where we control the artifacts.

04 · Stage 2

Standardization & ingestion plumbing.

Layer 1 adapters + Layer 2 transforms.

Once Stage 1's bronze tier is stable, we build adapters that read bronze and produce silver — FHIR R4 Bundles, profile-validated. Each adapter is a vertical slice owned end-to-end before moving to the next.

How we work
  • One adapter at a time, end-to-end (don't build five 70%-done adapters)
  • Test fixtures land alongside the adapter — minimum 3 inputs each
  • Schema-validation gate is automated; non-conforming Bundles are rejected at the silver boundary
  • LLM-bootstrapped crosswalks (e.g., Epic-table → FHIR-resource) are validated against ground truth and frozen as static files; never re-generated at runtime
Done means
  • Every source produces a profile-validated FHIR Bundle
  • CCDA → FHIR uses Microsoft FHIR-Converter (we don't reinvent)
  • Crosswalk decisions are documented per-source in docs/mapping-decisions.md
  • Re-running the pipeline on the same bronze input produces byte-identical silver output
05 · Stage 3

Harmonization layer.

Layer 3 — the differentiation work.

With clean silver Bundles per source, Layer 3 takes the cross-source merge problem on directly: identity resolution, code mapping, temporal alignment, dedup, conflict detection, Provenance emission. Sub-tasks ordered by dependency.

How we work
  • Sub-tasks ordered by dependency: identity → code map → temporal align → dedup → conflict → Provenance
  • Integration tests run on the showcase patient after every commit; the 5-artifact suite is the regression bar
  • Visual diff: before/after JSON for every merge — easy to inspect what the harmonizer actually did
  • UMLS lookups are deterministic; LLM use is bounded to the free-text-extraction sub-task and conflict-narration (build-time)
Done means
  • Showcase patient passes the full 5-artifact suite
  • Every gold-tier resource has at least one Provenance edge to silver
  • Conflicts are detected, not silently resolved — both records preserved
  • Quality scores are explicit (not learned), inspectable, defensible in narrative
06 · Stage 4

Vision extraction sub-layer.

PDFs → FHIR with page-level Provenance.

The vision extraction work begins as notebook exploration — testing prompts on real PDFs, iterating on the schema-constrained output format — and only after the prompts stabilize does it become production code with caching and Provenance emission.

How we work
  • Notebook-driven prompt iteration first; freeze the prompt before writing production code
  • Pydantic / instructor schemas are the contract — output validation is part of the test
  • Content-hash cache from day one: same PDF in, same extraction out
  • "Golden output" fixtures: one extraction per document type committed to the repo as the reference
  • Confidence threshold enforced — low-confidence extractions flagged, not silently shipped
Done means
  • The showcase lab PDF produces an Observation with the right LOINC, value, unit, date
  • meta.extension(.../source-locator) carries valid page+bbox
  • Cache invalidation works correctly on prompt-version change
  • Re-running extraction on cached input is a no-op (no LLM call)
07 · Stage 5

Application integration.

Sources panel + PDF page-bbox highlighting + AI bonus.

The patient-journey app already exists in production. Stage 5 wires it to the harmonized data and adds the new UI surfaces — the Sources panel for Provenance walking, the PDF viewer with bounding-box highlighting, the conflict side-by-side, the "what changed since last visit" diff. UI work is mocked against fake harmonized data first; real-data wiring follows once Layer 3 stabilizes.

How we work
  • Mock the harmonized data first; build UI without dependency on Layer 3 timing
  • Wire to real data via the existing Layer 4 SOF warehouse — no app-code changes for upstream changes
  • Demo rehearsal happens twice: once mid-stage, once before submission
  • Live URL deployable from a single command; no human-in-the-loop deploys near the deadline
Done means
  • Sources panel walks Provenance edges for every fact in the briefing
  • Clicking a PDF-derived fact opens the source PDF with bbox highlighted
  • End-to-end demo runs without manual intervention
  • AI-bonus angle ships as a clean narrative beat in the demo flow
08 · Stage 6

Submission packaging.

Narrative + design + repo + frozen URL.

The submission itself is a deliverable: a Phase 1 narrative PDF, design wireframes (these pages plus what we add for the actual narrative), a public GitHub repo, and a frozen live demo URL. Stage 6 is mostly assembly + final review; nothing in this stage should require new code.

How we work
  • Narrative drafted from the existing docs — EXECUTION-PLAN.md, DATA-AGGREGATION-LAYER.md, prior-art survey, strategic-options memo — not rewritten from scratch
  • Wireframes assembled from these design pages plus screenshots of the live app
  • Repo is cleaned for public view: README rewritten, licenses checked, attribution to Mandel prominent
  • Live URL frozen 24h before deadline; no deploys after the freeze
Done means
  • A judge reading the narrative cold arrives at the same one-paragraph summary we wrote in EXECUTION-PLAN.md §"How a competitor or judge would read this submission"
  • Public repo passes a "fresh-clone" reproducibility test
  • Live URL works, shows the showcase patient, demonstrates the Sources panel + PDF highlighting
  • Submission package uploaded by deadline; confirmation received
09 · Meta-workflow patterns

The threads that run across all stages.

Quality gates per stage

Each stage's "Done means" block above is the gate. Moving to the next stage is an explicit alignment moment — Chief reports done-status against the bullets, Blake confirms or pushes back. Skipping a gate is a real choice, made explicitly with a stated reason, never by accident.

Open questions surface immediately

When a decision is required, Chief surfaces it with options + recommendation + the cost of deferring. Blake answers; the answer goes into the relevant doc. We don't accumulate ten open questions and resolve them in a batch — that's how plans drift.

Cuts are explicit, not silent

When scope strains the deadline, the cut order is named in EXECUTION-PLAN.md §Risks. For Phase 1: SDoH first → AI-bonus polish → multi-source conflict count → AI-bonus core. Layer 3 harmonization is non-negotiable — without it the differentiation evaporates.

Sub-agents for parallel research

When research would otherwise stall the main thread, Chief spawns a focused sub-agent with a self-contained brief. The brief includes: what to find, where to write the output, what format, what NOT to do (e.g., copyright limits on quoting). Each sub-agent's output is reviewed and synthesized back into the active execution plan before next steps.

Provenance applies to our own work, too

Every architecture or strategy claim cites its source. When we say the Phase 1 rubric is 100 + 20 bonus, the source is the official EHI Ignite rules PDF. When we say no commercial vendor publishes EHI Export TSV ingestion, the source is the prior-art survey with its own caveats. When we say Mandel's redaction studio auto-suggests PII clusters, the source is the live-site walkthrough screenshot. This is the same provenance discipline we ship to clinicians; we apply it to our own narrative.

Public posture is the default

Repo is open from day one (MIT). Architecture docs are written for an external technical reader, not internal-only shorthand. The technical-deep-dive page is the document we'd hand to Mandel or any technical peer. This forces clarity earlier and makes attribution natural.

The submission is the byproduct, not the goal. The goal is a working harmonization layer that produces a demonstrably better unified patient record. The submission narrative falls out of having that thing. If we focus on the narrative first, we end up with a good story about a thin demo. If we focus on the thing, the story writes itself.
